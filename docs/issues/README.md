## 为什么是这 8 步（而非更细）

- **00 必须独立且第一**:基线红 → 无法判断后续红是不是自己引入的。
- **01 把「砍导航」和「立骨架」合一**:都只动 `TAB_IDS`/`TAB_REDIRECTS` 两个数组与容器，拆开反而碎。
- **02/03 拆开**:待办接数据 与 证据+跳转 是两套接口、两处 review 焦点，合并会超出单 PR 体积上限。
- **04/05 故意小而独立**:低风险，先合攒 CI 信心，且可与 02/03 并行不打架。
- **06 单列**:依赖后端缺失语义，前端先占位，风险点独立。
- **07 完全旁路**:配置/分发层，零架构改动，不与线 A 抢文件。

## 步骤漂移处理（重要）

每一步落地都可能让**后续步骤的预测、改动面、甚至依赖失效**。例如 STEP-01 改了 `TAB_IDS` 的组织方式后，STEP-05 的「放回 tab」做法可能要改写。处理规则:

1. **预测是活的**:STEP 文件的「结果预测」在它**真正开工前**复核一次，不是写完就冻结。
2. **每步完成后做下游复核**:在「实际反馈 → 下一步影响」里，**显式列出哪些后续 STEP 需要据此更新**（哪怕只是一句「STEP-03 的跳转选择器需改名」）。
3. **看板状态加注**:若某 STEP 因上游变化需要返工，在状态总览表把它标 🟠 待复核，并在该 STEP 文件顶部写一行「⚠️ 因 STEP-XX 变化待更新」。
4. **设计锁同步**:若某步发现 `design-lock-cn.md` 的冻结事实已不成立，在该 STEP 的「实际反馈」记录，并在 PR 里带 `[design-lock change]` 更新锁文档——不静默改。
5. **依赖未变不重写**:仅当上游确实改变了下游的前提时才动下游文件，避免无谓 churn。

> 一句话:**把「上游改动 → 下游待更新清单」当作每步的强制交付物**，让漂移显式、可追踪，而不是事后才发现对不上。
